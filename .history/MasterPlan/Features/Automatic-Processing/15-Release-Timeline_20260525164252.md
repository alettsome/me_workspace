## Deprecation Policy

### **What Will Never Be Removed**

- Core automatic processing (PDF → chunks)
- Database persistence
- Filesystem backup (dual-write)
- Local-first architecture

### **What May Change**

- Chunk size algorithm (non-breaking config)
- File filter patterns (non-breaking config)
- API response format (versioned)

### **What May Be Removed**

- Unused formats (if <1% usage)
- Legacy API endpoints (with deprecation notice)
- Redundant features (if better alternative exists)

**Deprecation Timeline**: 6-month notice before removal

## Version Numbering

**Scheme**: Semantic Versioning (MAJOR.MINOR.PATCH)

**MAJOR**: Breaking changes (API, schema, config)  
**MINOR**: New features (backward-compatible)  
**PATCH**: Bug fixes (no new features)

**Examples**:
- v1.0.0 → v1.0.1: Bug fix (extraction error handling)
- v1.0.1 → v1.1.0: New feature (web UI)
- v1.1.0 → v2.0.0: Breaking change (new chunk schema)
